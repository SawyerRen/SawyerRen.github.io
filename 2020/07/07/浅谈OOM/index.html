浅谈OOM的各种原因

发表于 2020-07-07 更新于 2020-07-11 分类于 jvm

heap space

在堆中创建的对象占用的内存超过了堆的最大内存，就会出现OOM。

GC overhead limit exceeded

GC回收时间过长会抛出OOM。过长的定义是，超过98%的时间都用来做GC并且回收了不到2%的堆内存。假如不抛出这个错误，那么GC回收清理的内存很快会被再次填满，迫使GC再次执行，会形成恶性循环。

出现这个错误时，可以看到错误信息：

说明GC垃圾回收几乎没有回收对象，会一直GC下去。这样就会报GC overhead limit exceeded。

direct buffer memory

写NIO程序经常使用ByteBuffer来读取或者写入数据，这时一种基于通道和缓冲区的I/O方式。它可以使用Native函数库直接分配堆外内存，然后通过一个存储在Java堆里面的DirectByteBuffer对象作为这块内存的引用进行操作。可以显著提升性能，避免了Java堆和Native堆中来回复制数据。

但是如果不断分配本地内存，堆内存很少使用，那么JVM就不需要执行GC，DirectByteBuffer对象就不会被回收，这时候堆内存重组，但本地内存可能已经使用光了，再次尝试分配本地内存就会OOM。
